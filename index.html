1 核心执行生命周期

了解了底层由谁构成后，我们来看执行流程。从启动进程到最终退出的宏观流程，核心思想是：所有的代码执行本质上都是同步的，异步只是任务的分发与回调的排队。

1

1. 顶层同步代码

主线程(V8)像复读机一样执行初始代码。遇到异步操作时（通过 Bindings 交给 Libuv）仅仅是“登记”，然后继续往下读。

2

2. 首次清空微任务

只要顶层代码包含 `Promise` 或 `nextTick`，在进入 Libuv 的事件循环之前，立刻强迫自己清空它们。

3

3. 事件循环流转

进入由 Libuv 维护的事件循环。拿出一个宏任务执行(交还给 V8)，**执行完瞬间立刻再次清空微任务**。

4

4. 休眠或退出

如果所有队列清空，且没有等待的 I/O（如未关闭的 Server 端口），进程结束退出。

3 任务队列对照表与优先级

微任务由 V8 引擎插缝执行，宏任务由 Libuv 按轮次排队。

相对优先级得分图

⚡ 微任务 (Microtasks)

由 V8 引擎维护，优先级最高
  • VIP
    process.nextTick()
    Node 环境特有，在当前操作完成后立刻执行，排在所有微任务最前面。
  • 高优
    Promise.then/catch/finally
    紧随 nextTick 之后执行。

📦 宏任务 (Macrotasks)

由 Libuv 维护，按事件循环阶段排队
  • Timers
    setTimeout / setInterval
  • Poll
    I/O 回调 (fs读写, 数据库, 网络请求)
  • Check
    setImmediate()
  • Close
    socket.on('close') 等清理任务

📖 知识速查：底层运转三大铁律

基于我们深入的探讨，这里为您提炼了最核心的架构理念与比喻，便于随时回顾。

铁律一：同步本质

V8 主线程永远只在执行当前的“同步代码”。不管是顶层逻辑还是 `setTimeout` 里的回调，在拿到 V8 执行的那一刻，它都是同步阻塞的。

所谓的“异步不会阻塞”，仅仅是因为遇到 I/O 时，它通过 Bindings 登记一下，把耗时任务丢给底层 Libuv 甚至 OS 去做了，主线程立即去读下一行。

🧹

铁律二：房间与走廊

清洁工隐喻：Node 像一个疯狂的清洁工。宏任务（Libuv分发）是“大房间”，微任务（V8维护）是房间外的“走廊”。

每打扫完一个大房间，他必定去走廊检查。只要走廊有垃圾（微任务），他就**一直扫直到彻底干净**，然后才进入下一个大房间（下一个宏任务阶段）。

铁律三：微任务饥饿

如果一直有微任务产生（例如在 `nextTick` 或 `Promise.then` 里递归调用自身），V8 主线程就会困在“清扫走廊”的过程中永远出不来。

这会导致底层事件循环（Libuv）完全被卡死，后续排队的宏任务（网络请求、定时器等）**永远不会被执行**，应用假死。
